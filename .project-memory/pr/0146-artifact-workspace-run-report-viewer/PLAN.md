## OPTION DECISION

### Decision: OPTION B — NARROW READ-ONLY REPORT API

A new `GET /runs/<run_id>/report` endpoint returning the complete run-report.txt, with provenance metadata, is required.

**Why Option A (existing detail packet) fails:**
- `detail.report_preview` is sliced to 500 characters at `report_content[:500]` (runtime_evidence.py L245). This is a fragment, not a complete report. Rendering this as a "report viewer" would be misleading.
- There is no truncation indicator — users cannot tell they are seeing only the first 500 characters.
- The roadmap explicitly requires reading "the report through an API, not an arbitrary file path" and showing the report content. A 500-char preview does not satisfy this.

**Why Option B satisfies the roadmap:**
1. A single new `GET /runs/<run_id>/report` route returns the complete run-report.txt with versioned JSON envelope.
2. The route reuses existing `run_id` validation and `runs_root` from server.py.
3. The response includes complete content, missing/truncated/unreadable states, and exact provenance fields.
4. No run persistence or report generation changes — only a new reader path.
5. Backward compatible — existing GET /runs/<run_id> detail.report_preview continues to exist unchanged.

### Report API contract

| Property | Value |
|---|---|
| Route | `GET /runs/<run_id>/report` |
| HTTP method | GET only |
| HTTP status | 200 with versioned JSON envelope |
| Content type | `application/json` |
| Version field | `ev_contract_version: "1"` (same as existing detail contract — additive, not breaking) |
| runs_root | Consistent with existing GET /runs and GET /runs/<run_id> — default to os.path.join(os.getcwd(), ".ariadne", "runs") |
| run_id validation | Reuse existing `_RUN_ID_RE = re.compile(r"^[a-zA-Z0-9_\-]+$")` from server.py |

### Response envelope

```
{
    "ev_contract_version": "1",
    "ok": true,
    "error": null,
    "run_id": "<run_id>",
    "content": "full report text...",
    "content_length": 12345,
    "truncated": false,
    "truncation_limit": null,
    "report_exists": true,
    "manifest_lists_report": true,
    "provenance": "run.json and manifest.json exist for this run_id. run-report.txt is listed in manifest files."
}
```

| State | ok | content | error | report_exists | manifest_lists_report |
|---|---|---|---|---|---|
| Complete report | true | full content | null | true | true/false |
| Empty report file | true | "" | null | true (file exists) | true/false |
| Missing report | false | null | "file_not_found" | false | false |
| Unreadable/malformed | false | null | OSError description | file exists but unreadable | depends |
| Invalid run_id | false | null | "invalid_run_id" | false | false |
| Unknown run | false | null | "run not found" | false | false |

**Route implementation design** (server.py):

The existing detail route is:
```
if method == "GET" and path.startswith("/runs/"):
```

The new report route must be placed BEFORE this check:
```
if method == "GET" and path == "/runs/<run_id>/report":
    # extract run_id from /runs/<run_id>/report
    # validate, read report, return JSON
```

Implementation approach: After extracting run_id, check if the remaining path suffix is `/report`. The cleanest approach is to parse the path segments:
```python
if method == "GET" and path.startswith("/runs/") and path != "/runs":
    parts = path.split("/")
    # parts = ["", "runs", "<run_id>"] or ["", "runs", "<run_id>", "report"]
    if len(parts) == 4 and parts[3] == "report":
        # report route
    elif len(parts) == 3:
        # existing detail route
```
