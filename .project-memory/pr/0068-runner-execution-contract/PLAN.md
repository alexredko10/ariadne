# PR 0068 — Runner Execution Contract Boundary Plan

## Goal

Plan the minimal runner execution contract boundary.

The contract should define how the existing mock application/run layer can hand a future execution request to a runner adapter and receive a deterministic execution result/status/evidence response.

This PR must remain contract-only.

It must not execute anything.
It must not introduce Docker.
It must not implement worker/orchestrator behavior.

## Architectural Thesis

The completed mock app loop creates and exposes deterministic task intake, context preview, mock run, and mock run status.

The next architectural boundary is the runner execution contract.

The runner should be a replaceable execution adapter boundary, not part of the application mock loop and not hardwired to Docker.

Docker agents are one future adapter implementation, not the contract itself.

The contract must support future execution modes such as:

- no-op/dry-run adapter
- local process adapter
- Docker agent adapter
- remote sandbox adapter
- human-gated adapter

But PR 0068 should only define the boundary, not implement those adapters.

## Contract Scope

The runner execution contract defines:

1. **RunnerExecutionRequest** — what the app layer sends to a runner adapter
2. **RunnerExecutionResult** — what the runner adapter returns
3. **Status values** — contract-level execution states
4. **Artifact reference** — how produced artifacts are reported
5. **Evidence record** — how execution evidence is reported
6. **Human approval boundary** — where human review sits
7. **Dry-run support** — the contract supports no-op/dry-run adapters

## Execution Request Contract

**Schema:** `schemas/runner-execution-request.schema.yml`

```yaml
schema_version: "0.1"

# RunnerExecutionRequest — what the application layer sends to a runner adapter.
#
# This is the boundary between the mock app loop (task-intake) and a future
# runner adapter.  The request carries all context the adapter needs to
# execute a run task.

# Required fields:
#   execution_request_id: string
#   run_id: string
#   task_intake_id: string
#   context_preview_id: string
#   requested_adapter: string (adapter kind identifier)
#   execution_mode: string (one of: "dry_run", "execute", "preview")
#   inputs: dict (execution inputs — adapter-specific structure)
#   constraints: list of string

# Optional fields:
#   expected_outputs: list of string
#   metadata: dict
#   approval: dict (approval state, if human approval is required)
```

**Key design rules:**
- `requested_adapter` is a string identifier, not an import path or class reference. The app layer doesn't know what adapter will handle the request — it just names the kind.
- `execution_mode` allows dry-run, full execute, or preview. This enables testing without real execution.
- `inputs` is a dict — the adapter defines its own input structure. The contract is a wrapper, not a prescriptive data schema.
- No absolute paths. No Docker-specific fields. No model/provider-specific fields.
- No timestamps — the request is stateless. Timestamps are in the result.
- JSON serializable.

## Execution Result Contract

**Schema:** `schemas/runner-execution-result.schema.yml`

```yaml
schema_version: "0.1"

# RunnerExecutionResult — what a runner adapter returns after execution.
#
# The result carries status, produced artifacts, and evidence from one
# execution request.  The adapter is responsible for populating this
# structure.

# Required fields:
#   execution_result_id: string
#   execution_request_id: string
#   run_id: string
#   status: ExecutionStatus
#   adapter: string (adapter kind identifier that produced this result)
#   artifacts: list of RunnerArtifact
#   evidence: list of EvidenceRecord

# Optional fields:
#   started: string (ISO 8601, adapter-provided)
#   completed: string (ISO 8601, adapter-provided)
#   errors: list of ErrorRecord
#   warnings: list of string
#   review_required: bool
#   next: string (suggested next step)
```

## Status Values

**Status values for `RunnerExecutionResult.status`:**

| Status | Description | Terminal |
|---|---|---|
| `accepted` | Adapter accepted the request but hasn't started | No |
| `running` | Execution is in progress | No |
| `completed` | Execution completed successfully | Yes |
| `failed` | Execution failed | Yes |
| `cancelled` | Execution was cancelled | Yes |
| `blocked` | Execution is blocked (e.g., pending human approval) | No |
| `requires_review` | Execution completed but requires human review | No |

These are future/runtime states that will be produced by runner adapters. Not all are used in the mock loop.

## Artifact Contract

**Artifact shape within `RunnerExecutionResult.artifacts[]`:**

```yaml
artifact_id: string
artifact_kind: string
relative_path: string (relative path, no absolute paths)
digest: string or null (SHA-256 hex, optional)
producing_step: string (step ID)
summary: string (human-readable summary)
```

No artifact files are created in PR 0068. This is a contract definition only.

## Evidence Contract

**Evidence shape within `RunnerExecutionResult.evidence[]`:**

```yaml
evidence_id: string
evidence_kind: string
summary: string
status: string (one of: "passed", "failed", "warning", "skipped", "not_run")
validation_ref: string or null (reference to the validation that produced this)
artifact_ref: string or null (reference to a produced artifact)
```

This aligns with the existing `VerificationEvidence` entity in the Core runtime (PR 0047) but is at the runner adapter boundary, not the Core runtime layer.

## Human Approval Boundary

The contract represents human approval through:
- `RunnerExecutionRequest.approval` — optional dict indicating approval state
- `RunnerExecutionResult.status: "blocked"` — execution is blocked pending approval
- `RunnerExecutionResult.review_required: bool` — execution completed but needs review
- `RunnerExecutionResult.status: "requires_review"` — execution completed with review-required result

PR 0068 does NOT implement:
- Approval UI
- Notification behavior
- Policy engine
- Approval workflow
- Review routing

These belong in future runner adapter and conductor work.

## Relationship to Docker Agents

- Docker agents are future runner adapters.
- Docker is not the execution contract.
- PR 0068 does not introduce Docker files or commands.
- PR 0068 does not assume Docker is available.
- A future Docker adapter must conform to this contract.
- The `requested_adapter` field in the request schema allows adapter-specific identifiers like `"docker-coder-v1"` without the contract knowing about Docker.
- The existing `docs/DOCKER_AGENTS_MEMORY.md` describes current manual agent setup. This contract is the first step toward replacing that manual flow with a programmatic adapter boundary.

## Open Questions

1. **Should the contract include a `RunnerAdapterCapability` descriptor?** **Decision:** No — deferred to a future PR that implements the first adapter. The capability descriptor would describe what adapters can do (supported execution modes, required inputs, etc.). This is useful but not required for the boundary definition itself. ADR 0010 can note that capability descriptors are a future concern.

2. **Should `execution_mode` be a string enum or a fixed set?** **Decision:** String with documented allowed values (`"dry_run"`, `"execute"`, `"preview"`). This allows future adapters to define additional modes without schema changes. The schema documents the initial three.

3. **Should `inputs` (the adapter-specific input dict) have a minimal required structure?** **Decision:** No. The contract is a wrapper. Each adapter defines its own input schema. The contract ensures the wrapper fields are consistent across all adapters. This is consistent with the architectural thesis that the runner is a replaceable adapter.
